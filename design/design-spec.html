12. Responsive Rules
Mobile-first. The portal is used on mobile. The admin dashboard is mostly desktop.
BreakpointAdmin DashboardClient Portal
Mobile (<768px)Sidebar collapses to top header with hamburger. Stats: 2-col grid. Main content: single column. Table views become card stacks.Top header + bottom tab nav (4 tabs). Single column content. Cards stack vertically. Download buttons full-width.
Tablet (768-1024px)Sidebar stays visible (narrowed to 200px). Stats: 2-col or 4-col. Activity+clients: stacked vertically.Top nav visible (no bottom tabs). 2-col project grid.
Desktop (‚â•1024px)Full sidebar 220px. Stats 4-col. Activity+clients 2:1 split. Full kanban view.Full top nav. 2-3 col project grid depending on count.

Touch Targets

All interactive elements must be minimum 44√ó44px touch target on mobile. This includes:

  • Nav items in the portal bottom bar ‚Äî use py-3 + px-4 minimum
  • Document download buttons ‚Äî full-width on mobile
  • Client list rows ‚Äî make entire row tappable, not just the text
  • Color swatches in onboarding ‚Äî minimum 40√ó40px
13. Missing Screens / Gaps
These screens are linked but not built. Prioritized by user impact.
PriorityScreen / FeatureWhy NeededComplexity
P0 /clients/[id] ‚Äî Client Detail Dashboard links to client detail pages (/clients/1 etc). Currently 404. Linked from multiple places. Blocks core workflow. Medium ‚Äî needs: client header, project list, document list, activity log for this client, message thread link
P0 Portal branded login screen First impression when client clicks magic link. Currently unclear ‚Äî should show firm branding, not generic login. Low ‚Äî reuse ClientPortalShell wrapper
P1 Onboarding success screen After step 4, user needs positive reinforcement and clear next steps. "Portal is live" moment. Low ‚Äî interstitial before /dashboard redirect
P1 Empty states (all list pages) New users see empty Clients, Projects, Documents pages. Without empty states they're confused about what to do next. Low ‚Äî one shared EmptyState component, used 4x
P1 /activity ‚Äî Full activity log Dashboard links to "View all activity" ‚Üí 404. This is the key engagement signal (when did client read proposal?) Low ‚Äî table of Activity records, same pattern as Documents
P2 /projects/[id] ‚Äî Project Detail Clicking a project card should go somewhere. Currently no project detail page ‚Äî blocks the document upload flow within a project context. Medium ‚Äî tabs: Overview, Documents, Messages, Team
P2 /terms and /privacy Linked from signup footer ‚Äî 404. Required for credibility and legal compliance. Very low ‚Äî static content pages
P2 Notification panel Mobile header has bell with badge count but tapping it does nothing. Needs slide-down panel. Low ‚Äî slide panel listing activity events
Critical for coder: Focus P0 items first (client detail page + portal login). These complete the core loop: invite client ‚Üí client logs in ‚Üí firm sees activity ‚Üí firm follows up. Without these, the app has holes in the core workflow.
